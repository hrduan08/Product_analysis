# YouTube & Reddit 反馈抓取 Demo 需求文档

## 1. 背景与目标
- 验证自动抓取用户在 YouTube、Reddit 两大内容平台上对指定产品的公开反馈信息的可行性。
- 搭建可扩展的跨平台采集与展示框架，为后续引入 Amazon 等平台奠定基础。
- 快速交付可演示的功能：输入关键词 → 选择平台 → 返回对应的视频或帖子列表 → 支持打开原链接。

## 2. 用户场景
- 产品经理或开发者输入品牌/产品名称，获取最新测评视频或社区讨论帖。
- 市场/运营人员定期查询，收集团队关注产品的动态与用户反馈。

## 3. 功能范围
### 3.1 实时查询入口（Web）
- 前端单页应用提供平台切换（YouTube / Reddit）+ 关键词输入框。
- 支持历史查询（最近 5 条，区分平台）和清空操作；输入为空时禁用搜索按钮。
- 调用统一接口 `GET /api/search?platform=...&query=...`，后端根据平台派发到对应 Provider。
- 前端展示卡片：标题、作者/频道、发布时间、抓取时间、平台标签、跳转链接；YouTube 卡片显示缩略图/观看量，Reddit 卡片显示子版块/得分/评论链接。
- 状态体验：Skeleton、空态插图、错误提示、复制链接 Toast、分页滚动回顶部。

### 3.2 平台抽象
- 后端统一 `FeedbackItem` 数据模型，Provider（YouTube/Reddit）实现 `search({ query, cursor, limit })` 接口。
- 保留 `platform` 字段和 `pageInfo.nextCursor`，方便前端复用分页逻辑。
- 速率控制：按 `platform + keyword` 最小间隔（默认 30 秒，可配置）。

### 3.3 搜索控制台配置（登录用户）
- 入口：登录用户在导航中进入“搜索配置”页（与实时查询页区分）。首屏提示配置步骤与当前配额/剩余额度。
- 平台勾选：
  - 提供 **YouTube、Reddit、X（Twitter）与 Facebook** 四个勾选项，默认仅勾选 YouTube（Reddit 需用户手动开启）。
  - 采用多选控件，点击勾选/取消即刻生效（无需“保存”按钮）；界面应提示“选择的内容平台将即时应用于后续自动搜索”。
  - 至少保持 1 个平台被勾选；若用户将所有选项取消，立即弹出提示并恢复上一次有效状态。
  - 后端对平台枚举做校验，非法值返回 400。
- 关键词管理：
  - 标题或描述直接说明“最多可添加 3 个关键词”；当列表已满 3 条时，“添加关键词”按钮置灰且禁用，提示文案保持可见。
  - 每个关键词长度 2–60 字符，支持中英文及常用符号；提交前去除首尾空格并做不区分大小写的去重处理。
  - 支持编辑与删除；删除后按钮恢复可用。
- 搜索时间配置：
  - 支持设置每天最多 3 个执行时间（24 小时制，精度到分钟），UI 建议使用时间选择器。
  - 至少设置 1 个时间点；时间列表自动排序且禁止重复。
  - 保存时生成 cron 表达式或任务配置，写入用户订阅记录。
- 邮件接收配置：
  - 默认读取当前注册账号邮箱并回填，用户可以修改，但仅允许维护 1 个邮箱地址。
  - 邮箱格式校验通过再允许提交；可提供“发送测试邮件”按钮（后续实现）。
- 保存与状态反馈：
  - 表单区域实时校验，所有字段满足条件后才能提交。
  - 保存成功后提示“配置已保存，将在下一执行时间开始自动搜索”。
  - 后端保存接口应具备幂等性，重复提交覆盖旧配置，并返回结构 `{ platforms, keywords, schedules, email, nextRunAt }`。
- 数据约束：
  - 每个用户在数据库层面存储配置 JSON 或拆分多表，需持久化关键词顺序、平台集合、时间点、邮箱、上次执行时间。
  - 若试用/套餐存在限制（如关键词数上限），在该界面实时校验并提示升级指引。
- 空态引导：
  - 初次进入时展示配置指南，包含“选择平台”“添加关键词”“设置时间”“确认邮箱”四个步骤，可选提供快捷按钮“一键导入推荐关键词”。

### 3.4 定时搜索与自动推送

#### 3.4.1 功能目标
- 用户在“搜索配置”中设定的关键词、平台、执行时间与通知邮箱，系统自动在指定时间调用外部 API 抓取增量内容，并以邮件形式推送给用户。
- 仅在关键数据源（当前为 YouTube，如果 Reddit 凭证缺失则自动禁用）可用时执行任务；避免因第三方服务异常导致错误提示或账号封禁。

#### 3.4.2 用户体验与配置项
- **平台选择**：
  - 页面实时显示哪些平台已启用；若后台检测到某平台的凭证缺失（例如 Reddit 配置为空），前端只展示 YouTube，并在提示区说明“Reddit 暂未启用”。
  - 用户之前勾选的已禁用平台会在刷新后自动去除（保持数据一致性）。
- **关键词管理**：同 3.3 中的规范（最多 3 个，去重校验）。
- **执行时间**：
  - 支持添加 1~3 个每日执行时间，默认精度到分钟。
  - 前端提示用户“系统会在所选时间附近（例如 5 分钟内）执行，以避免高峰时段拥堵”。
  - 若后台队列繁忙，系统可能延迟执行并在邮件/配置页标注“延迟 X 分钟”。
- **通知邮箱**：默认使用注册邮箱，支持改成其他地址，每个账号仅保存一个；成功保存后提示“配置已保存，将在下一次执行时生效”。
- **时区**：用户可修改时区，系统按照该时区计算执行时间；默认 `Asia/Shanghai`。
- **状态展示**：
  - 配置页顶部展示“下一次自动搜索时间”“上次执行时间”；刷新页面或执行完任务后实时更新。
  - 任务执行失败时可在配置页提醒（例如“最近一次执行失败，请检查邮箱或稍后重试”）。

#### 3.4.3 运行规则
- **仅保留用户自助搜索任务**：部署时依赖用户自助配置；旧的全局默认关键词 Cron 已移除，避免与当前 SaaS 模式混淆。
- **平台启用判断**：
  - 后端启动时检测 Reddit 环境变量是否齐全；缺失时仅注册 YouTube Provider，并在配置页返回 `supportedPlatforms=['youtube', 'x', 'facebook']`（其中 `x/facebook` 为占位，下阶段接入）。
  - 用户保存配置后，后台会过滤掉未启用的平台，确保不会触发无效调用。
- **调度频率**：
  - 后端统一的 `USER_SEARCH_CRON_SCHEDULE` 只负责“唤醒调度器”（建议默认 `*/1 * * * *` 每分钟检查一次）。真正执行哪一条任务取决于配置中的 `slots` 与 `timezone`，用户在前端调整时间后立即生效，无需重启服务或等待固定波段。
  - 若队列过于繁忙，系统可延迟执行并在 `notify_jobs` 中记录延迟原因，同时在配置页提示“上次执行延迟了 X 分钟”。
- **调用频率与队列**：
  - 当前 MVP 阶段按配置时间直接执行；后续将引入限流队列，控制每分钟调用量，避免外部 API 超限。
  - 同一时间段多个用户任务集中时，会按用户 ID 顺序串行调用，减少瞬时压力。若队列过长，将延迟执行并写入日志。
- **增量抓取**：
  - 每次执行时调用 Provider（仅 YouTube）获取查询结果，将数据 upsert 至 `feedback_items` 并依据 `first_seen_at` 判断新增内容。
  - 若无新增内容且用户未开启“空邮件提醒”，则跳过邮件发送。
- **邮件推送**：
  - 使用 SendGrid（或 stub）发送 HTML + Text 报告，按平台/关键词分组列表化展示。
  - 邮件中包含执行时间、上次成功时间、增量条数，以及每条内容的标题、作者、发布时间、链接等。
- **告警与退避**：
  - 调用外部 API 失败时记录日志，并通过 Slack Webhook（若配置）发送告警；连续失败会暂停该用户任务，并在配置页提示。
  - 邮件或 Slack 请求异常（如 `ECONNRESET`）时会重试 1 次，仍失败则记录 `notify_jobs` 并提示用户。

#### 3.4.4 数据存储与追踪
- `user_search_configs`：保存每个用户的配置（关键词、平台、执行时间、通知邮箱、时区、下一次执行时间、最近执行时间）。
- `notify_jobs`：记录任务执行历史（成功/失败、发送数量、错误信息、上下文），便于追溯问题。
- `feedback_items`：存储抓取的原始数据，用于增量判断与邮件内容。

#### 3.4.5 失败与异常处理
- 缺少有效平台或关键词/执行时间为空时，后台跳过任务并更新 `notify_jobs` 状态为 `partial/success`，同时在日志里输出原因。
- 邮件发送失败会在 `notify_jobs` 中记录 `error_message` 并触发 Slack 告警；用户配置页会提示“最近推送失败，请检查邮箱或联系支持”。
- 外部 API 超时或限制时，在日志中记录 `rateLimitExceeded` 等错误，并计划在后续版本加入退避和限流机制。

### 3.5 后台自动抓取能力
#### 3.5.1 功能目标
- 长期、定时地抓取用户关注的关键词在各平台的最新内容，并在后台存储，以便后续邮件推送、后台查看与数据分析。
- 在没有任何用户配置的情况下，支持运营团队维护全局抓取任务（可选）。

#### 3.5.2 功能要素
- **抓取频率**：调度器默认每分钟唤醒（可配置），若 `next_run_at` 到期则立即执行；允许运营自定义唤醒频率。
- **数据来源**：当前接入 YouTube；若 `.env` 中配置 Reddit 凭证，则自动启用 Reddit Provider。凭证缺失时系统自动禁用对应平台。
- **关键词来源**：
  - 用户自定义：来自 `user_search_configs`（详见 3.4）。
  - 运营全局配置（可选）：`CRON_*` 环境变量支持维护默认关键词、平台、通知邮箱。
- **增量判断**：抓取结果写入 `feedback_items`，以 `(platform, external_id)` 去重，通过 `first_seen_at` 判断新增条目。
- **异常处理**：抓取或邮件发送失败时记录 `notify_jobs`，触发 Slack/Webhook 告警，并在前端提示用户。
- **日志与追踪**：运营可在后台（后续增强）根据 `notify_jobs` 查看任务执行情况，必要时手动重试。

#### 3.5.3 数据存储要求
- `feedback_items`：记录抓取内容（标题、作者、链接、发布时间、观看量等），支持增量查询。
- `notify_jobs`：记录每次任务执行结果（状态、增量条数、邮件 messageId、错误信息、上下文）。
- `user_search_configs`：保存用户配置（平台、关键词、执行时间、邮箱、时区、下一次执行时间）。

#### 3.5.4 用户体验
- 普通用户：通过“搜索配置控制台”设置关键词、执行时间和通知邮箱；系统仅按用户配置执行。
- 运营人员：可通过 `.env` 管理默认抓取任务（可选），并查看 `notify_jobs` 日志（未来提供后台界面）。
- 邮件接收方：收到增量摘要邮件，可点击跳转到原平台查看详情。

#### 3.5.5 成功标准
- 每次任务执行后必须更新 `next_run_at`，确保下一次按时运行。
- 抓取失败率低于 2%，遇到外部 API 429/500 等错误时需立即告警。
- 邮件内容与 `feedback_items` 中的增量数据一致，数据准确。
- 用户设定的执行时间与邮件实际发送时间误差控制在 1 分钟内（受限于调度频率）。

#### 3.5.6 后续扩展（可选）
- 引入队列/限流机制，控制请求速率并提升稳定性。
- 扩展更多平台（Twitter/X、Facebook、Amazon、知乎等）。
- 提供后台界面查看全局抓取数据、手动触发重试、管理默认任务。
- 支持多渠道通知（Slack、Webhook、短信等）。

#### 3.5.7 YouTube API 调用优化策略
为在有限配额下支撑更多用户并避免封号，系统从“套餐约束、抓取策略、去重缓存、频率控制、多凭证扩容”五个维度进行组合优化：

1. **任务调度与套餐设计**
   - 付费体系仅保留三档：①新注册账号自动进入 7 天试用期，权益为“最多 1 个关键词 + 每日 1 个执行时间”；②正式付费会员权益为“最多 2 个关键词 + 每日 2 个执行时间”；③非会员（试用到期或付费过期）无法创建任何定时搜索，直到重新付费。
   - 依据套餐权益在后端计算“理论日消耗”，以 `search.list(100 units) + videos.list(~1 unit)` 估算：试用用户约 101 units/天，正式付费会员约 404 units/天。注册或升级前若计算结果会使平台总预算超标，则返回“目前配额不足，暂不支持新增订阅”提示。
   - `user_search_configs` 记录 `next_run_at` 与 `last_notified_at`。当用户修改执行时间后，只更新下一次触发点，等到新的 `next_run_at` 到期才真正执行，避免短时间内重复调用。

2. **配额友好的抓取策略**
   - 每次任务仍按“每个关键词一次 `search.list`”执行，但在送达邮件前会收集本轮需要通知的所有 `videoId`，按最多 50 条一批调用 `videos.list`，确保点赞/观看等统计字段一次性更新，避免对相同视频重复请求。

3. **缓存与去重**
   - `feedback_items` 保存 `first_seen_at` 与最近一次刷新时间，邮件只推送 `first_seen_at > last_notified_at` 的新增条目，确保用户只收到增量提醒，减少重复内容带来的不必要查询。

4. **控制 API 调用频率**
   - 调度器仍以 `*/1 * * * *` 唤醒，但在挑选任务时对 `next_run_at` 加入 0–30 秒随机抖动，降低同一秒内的大批量突发调用。
   - 以“API 凭证（API Key / OAuth Client）”为粒度维护速率计数器，每个 Key 独立记录 `search_calls_per_minute` / `videos_calls_per_minute`。计数器可通过内存 + 定时清零或 Redis `INCR/EXPIRE` 实现滚动窗口。
   - 初始阈值：`SEARCH_MAX_PER_MINUTE = 5`、`VIDEOS_MAX_PER_MINUTE = 15`。当某个 Key 在当前分钟内达到阈值，本分钟不再为其派发任务，调度器尝试切换到下一套 Key；若全部 Key 同时触顶，则将任务排队到下一分钟，并在 `notify_jobs` 记录“因限频延迟”的原因。
   - 任务执行顺序仍按 `next_run_at` 排序。调度器依次挑选“剩余额度充足且速率未触顶”的 Key，触发 `search.list` 与 `videos.list` 时同步自增计数；若需要多批 `videos.list`，需预留足够计数（例如 3 批即 +3）。失败重试也必须重新校验计数，避免重试反而触发限频。
   - 除速率外需记录“每日累积配额 = search×100 + videos×1”。单个 Key 若累计达到 90%（约 9,000 units），立即停止给它分配新任务，剩余任务切换到其他 Key，避免在误差情况下踩到 10,000 单位硬上限。
       - `quotaExceeded` / `userRateLimitExceeded` 时将该 Key 标记为冻结状态，直到下一分钟或次日重置，并写入日志与告警，方便排查。

5. **平台扩容与多凭证**
   - 预置多套 Google Cloud 项目或 API Key，将用户按哈希或注册顺序分桶到不同凭证池，分摊配额负载。
   - 通过本地计数或监控系统追踪每套凭证的日消耗。当单个 Key 的累计用量接近 70% 时提前发出预警，提示运营在配额耗尽前添加新 Key 或调整调度策略；真正达到 90% 时自动停止分配，确保整体可用性。

### 3.6 多渠道实时通知（用户可自定义飞书）
#### 3.6.1 背景与目标
- 邮件通知虽可靠，但很多团队希望在飞书群中实时同步抓取结果，便于讨论。
- 允许每个用户在“搜索配置”中自定义通知渠道（邮件/飞书），并填写自己的飞书机器人 Webhook，实现灵活推送。
- 后续仍可扩展钉钉、企业微信等渠道，但首个版本聚焦飞书。

#### 3.6.2 用户故事
1. 作为订阅用户，我希望将抓取摘要推送到自己团队的飞书群，而不是统一的公共群。
2. 作为用户，我可以在搜索配置里选择通知方式并填写自己的飞书机器人 Webhook，无需联系平台方配置。
3. 作为产品/运维，我需要知道每个任务是否成功通知飞书；若 Webhook 失效，应提示用户检查并回退到邮件。

#### 3.6.3 功能规格
- 搜索配置页新增“通知方式”模块：
  - 默认勾选“飞书”，用户可以切换为 `邮件` 或 `邮件+飞书`；如果只保留邮件，取消飞书勾选即可。
  - 选择飞书时只需要填写 Webhook URL（Secret 不再展示/必填），并可点击“发送测试”按钮验证；若用户的飞书机器人启用了签名，需要他们在飞书端关闭或后续版本再扩展支持。
  - 每个配置记录：`notifyChannels: ['feishu']`（默认） + 可选 `email`、以及 `feishuWebhook`。
  - 提示指引：“如何在飞书群创建自定义机器人并复制 Webhook”。
- 后端保存时校验：
  - Webhook 必须匹配 `https://open.feishu.cn/open-apis/bot/...`，并以密文存储。当前版本不再要求 Secret；若用户在飞书端启用了签名，需要他们关闭或等待后续版本支持。
  - 提供“发送测试”接口：用户保存前可点击测试，若飞书返回非 0 状态，显示错误原因。
- 调度触发：
  - 与邮件相同，仅在有增量时发送；若用户选择“邮件+飞书”，两个渠道都发送。
  - 飞书发送失败时：记录 `notify_jobs` 失败原因，同时提示用户检查 Webhook。若配置了邮件作为备用，可以 fallback 邮件并标记部分失败。
- 消息格式：
  - 标题：`【定时摘要】关键词 / 平台`。
  - 包含执行时间、增量条数、前几条详情（标题、作者、链接），并附“查看全部”链接。
  - 消息统一采用文本或卡片格式，通过用户提供的 Webhook 直接推送。

#### 3.6.4 安全与限频
- Webhook 以密文存储，前端仅显示“已配置”；如需更换，重新输入 URL 并保存。
- 限频：单用户飞书推送默认 ≤1 条/任务/触发，以 `notify_jobs` 记录耗时和响应码；若飞书返回 429，延迟重试并提示用户。
- 测试时可采用 30 秒限流，防止用户频繁点击测试导致封禁。

#### 3.6.5 成功标准
- 配置成功率 ≥ 95%，有错误时能提示具体原因（例如 Webhook 域名不符、飞书返回签名错误）。
- 飞书推送成功率 ≥ 98%，平均延迟 < 5 秒。
- 用户可以在配置页随时切换/关闭飞书，不影响邮件通知。

### 3.7 用户反馈中心
#### 3.7.1 目标
- 提供极低门槛的“提交反馈”浮窗，用户可快速描述问题、附截图并提交。
- 后端无需落库，直接将反馈内容推送到指定的飞书自定义机器人，团队在群聊中即时查看。

#### 3.7.2 用户流程
1. 登录或未登录用户都可在页面右下角看到“提交反馈”浮窗。
2. 表单字段：
   - 反馈类型（遇到问题/功能建议/其他，必填）。
   - 详细描述（必填，≥10 字，提示用户写明重现步骤）。
   - 标题、联系邮箱、截图（至多 5 张）均为可选。
   - 系统默认附带诊断信息：URL、浏览器 UA、语言、网络状态、最近 20 条前端日志 / API trace、用户 ID 或匿名 ID。
3. 用户点击“提交反馈”后，本地生成成功提示；后台会把文本、附件 URL、诊断 JSON 组装成飞书机器人消息推送到指定群聊。
4. 团队成员在飞书群内即可收到卡片消息，包含描述、截图链接、诊断展开信息；如需进一步追踪，可在飞书内回复或手动登记。

#### 3.7.3 功能规格
- 截图支持 PNG/JPG，单张 ≤5MB，共 5 张；上传时先写入对象存储，再把 URL 放入消息。
- 诊断信息示例：
  - `client`: URL、UA、language、online 状态、屏幕尺寸。
  - `logs`: 最近 20 条前端日志（包含 level、message、timestamp）。
  - `apiTraces`: 最近 10 次 API 调用。
  - `session`: userId 或匿名 ID、关联请求 ID。
- 成功标准：
  - 提交失败率 < 1%，飞书消息延迟 < 5 秒。
  - 诊断信息随反馈发送的覆盖率 ≥ 90%，显著缩短排查时间。

## 4. 非功能需求
- **可维护性**：数据源适配层独立实现，遵循统一接口，增加平台时无需改动前端逻辑。
- **性能**：单次查询响应时间 < 3 秒（包括 OAuth 取 token），分页请求保持 < 2 秒。
- **可靠性**：对关键错误（401/403/429/5xx）进行日志记录与友好提示，必要时触发 Token 刷新。
- **安全性**：所有外部密钥（YouTube、Reddit）通过环境变量注入，不在前端暴露；若使用代理，需在 `.env` 设置。

## 5. 技术方案
### 5.1 前端
- 技术栈：Vite + React + TypeScript。
- 组件：平台切换、搜索栏（含历史记录）、结果列表、分页、状态提示、Toast。
- 将 API 地址写入配置（`.env` 或构建时变量），支持 `VITE_API_BASE_URL`。

### 5.2 后端
- 技术栈：Node.js + Express。
- 路由：统一使用 `GET /api/search`，参数：
  - `platform`（`youtube`|`reddit`），必填。
  - `query`，必填。
  - `pageToken`/`after`，可选，用于分页。
- 职责：参数校验、速率节流、OAuth Token 管理（Reddit）、调用外部 API、整合数据。
- 缓存（可选）：按 `platform + query + pageToken` 维度缓存，默认关闭，可通过配置开启。

### 5.3 数据模型
```json
{
  "platform": "youtube",
  "keyword": "string",
  "id": "string",                // YouTube: videoId, Reddit: post id
  "title": "string",
  "channelTitle": "string",      // 或 Reddit: author
  "publishedAt": "ISO8601",
  "thumbnailUrl": "string|nullable",
  "url": "string",               // 外部链接或视频地址
  "permalink": "string",         // 平台内部链接（Reddit 专用，可为空）
  "viewCount": "number|nullable",
  "score": "number|nullable",    // Reddit 帖子得分
  "subreddit": "string|nullable",
  "fetchedAt": "ISO8601"
}
```
- 每条记录应附带 `pageInfo`：`resultsPerPage`、`totalResults`（若无则估算）、`nextPageToken`/`prevPageToken` 或 `after`/`before`。

## 6. 外部依赖与配置
- 必需：
  - Google Cloud 项目与 YouTube Data API Key（服务配额需监控）。
  - Reddit 开发者应用（script 类型）提供的 `client_id`、`client_secret`、授权账号的 `username`、`password`。
- 可选：Redis（如需跨进程缓存）。
- 环境变量：
  - `YOUTUBE_API_KEY`
  - `REDDIT_CLIENT_ID`
  - `REDDIT_CLIENT_SECRET`
  - `REDDIT_USERNAME`
  - `REDDIT_PASSWORD`
  - `PORT`
  - `CACHE_TTL_SECONDS`（默认 600，可留空禁用）
  - `QUERY_THROTTLE_SECONDS`（默认 30）
  - `HTTP_PROXY` / `HTTPS_PROXY`（如需代理访问）

## 7. 错误与日志
- 记录每次请求耗时、平台与关键词、响应状态和外部 API 错误码。
- YouTube/Reddit 返回 401/403 时自动刷新 Token 或提示检查 Key/凭证。
- 超过配额或速率限制时记录警告日志，并在响应中提示用户稍后再试。

## 8. 未来扩展预留
- 数据模型保留 `platform` 字段，继续扩展 Amazon、Twitter 等平台。
- 抽象数据源接口：`search({ query, pageToken, limit })`，不同平台独立实现。
- 后端保留任务队列接口，为未来接入情感分析、告警通知、数据导出做准备。

## 10. 商业化与会员付费（阶段 11 需求补充）

### 10.1 会员层级与权益
| 会员类型 | 收费/权益周期 | 价格 | 搜索配置权益 | 备注 |
| --- | --- | --- | --- | --- |
| 新注册试用 | 注册起 7 天 | 免费 | 最大 1 个关键词 + 每日 1 个执行时间；仅限 YouTube | 试用到期自动降级为“非会员” |
| 月会员 | 30 自然日 | 9 元/月 | 最大 2 个关键词 + 每日 2 个执行时间；支持多渠道通知（邮箱/飞书） | 付款后立即生效，顺延 30 天 |
| 年会员 | 365 天 | 99 元/年（折合 8.25 元/月） | 同月会员权益；优先支持更多 API 调度额度 | 用户可随时续费/升级 |
| 非会员（试用或付费到期） | - | - | 无法配置定时搜索；保留历史记录只读 | 需重新购买套餐才能恢复 |

- 套餐与账户状态映射：`trialing` → `active_monthly` / `active_yearly` → `past_due` / `expired`。
- 正式会员到期前若再次下单，同一计划的剩余天数顺延；试用用户支付后立即转为对应付费状态。

### 10.2 订阅生命周期
1. 用户完成注册 + 邮箱验证后进入 7 天试用，页面显示剩余天数和“升级会员”入口。
2. 在试用或付费有效期内，用户可配置关键词/时间。系统根据状态实时校验上限（试用 1/1、正式 2/2）。
3. 试用到期或付费过期时：
   - 自动暂停所有定时任务；
   - 搜索配置页展示“会员到期”提示，并仅允许查看历史记录；
   - 发送“到期提醒”邮件/飞书，附带付费入口。
4. 用户在“会员中心”选择套餐（默认展示月/年两档）→ 选择支付方式（微信支付 / 支付宝）→ 生成订单 → 完成支付 → 订单回调确认 → 更新 `plan_id` / `plan_expire_at`。
5. 付款成功后立即恢复定时任务，并重新计算 `next_run_at`。若订单失败或超时，则保持原状态。

### 10.3 支付渠道与体验要求
- **支持渠道**：微信支付（Native/JSAPI 二选一，优先 Native 二维码）与支付宝电脑网站支付，用户可任意选择。
- **支付流程**：
  1. 用户在前端选择套餐与支付方式后调用 `POST /api/payment/orders` 创建订单；后端生成唯一 `out_trade_no`，写入 `payment_orders`。
  2. 后端根据渠道调用微信/支付宝下单接口，返回二维码链接（Native）或支付跳转链接（支付宝）；前端展示二维码或自动打开支付页。
  3. 支付完成后，渠道推送异步回调到 `/api/payment/wechat/notify` 或 `/api/payment/alipay/notify`，系统校验签名、金额与订单号。
  4. 校验成功则将订单状态置为 `paid`，写入 `payments`/`invoices`，并调用会员升级逻辑；若失败/超时，则更新为 `failed` 或 `expired`。
  5. 前端轮询 `GET /api/payment/orders/:id` 或通过 WebSocket/Server-Sent Events 获取最新状态，在 5~10 秒内刷新界面。
- **订单状态**：`pending` → `paid` / `failed` / `expired` / `canceled`。支付过程中支持“重新拉起支付”，但需复用同一个订单号。
- **退款/回滚**：暂不提供在线退款，但需要在后台保留手动退款接口（更新订单并撤销会员权益）。

### 10.4 数据结构与接口要求
- `plans` 表新增两条记录：`MONTHLY_BASIC`（9 CNY、30 天）、`YEARLY_BASIC`（99 CNY、365 天）。
- `user_subscriptions` / `users` 表字段：
  - `trial_started_at`、`trial_ends_at`（注册时自动写入）；
  - `plan_id`、`plan_expire_at`、`status`（`trialing`/`active`/`past_due`/`expired`）；
  - `plan_source`（wechat / alipay / manual）。
- 关键接口：
  - `GET /api/app/membership`：返回会员状态、剩余天数、可用权益。
  - `POST /api/payment/orders`：创建订单。
  - `GET /api/payment/orders/:id`：查询订单状态。
  - `/api/payment/wechat/notify`、`/api/payment/alipay/notify`：异步回调。
  - `POST /api/app/membership/refresh`：回调成功后调用，用于刷新前端缓存。

### 10.5 提醒与运营动作
- 试用到期提醒：提前 24 小时推送邮件/飞书，含套餐对比表和快速支付入口。
- 付费会员到期提醒：提前 5 天和当日两次提示；若到期仍未续费，进入 `past_due` 并暂停任务。
- 支付异常提醒：订单 `failed` 或回调校验未通过时，触发系统告警（飞书）并提示用户重新支付。
- 定期推送“定时任务调用量”或“配额紧张”提示，引导用户升级或减少配置。
